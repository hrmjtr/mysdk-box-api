# 再実装ロードマップ(共通)

どの言語で再実装する場合も、同じ順序で進められるように
共通のステップを定義する。一度にすべてを実装せず、
**各ステップの完了条件を満たしてから次に進む**。

言語ごとの具体的なコードとコマンドは、各言語の
`docs/03-reimplement.md` がこのロードマップに沿って解説している。

## なぜこの順序か

- **最初に API を「実装なしで」理解する**(Step 0)。
  クライアントを書きながら API の仕様も学ぼうとすると、
  問題が起きたときに「API の理解が間違っているのか、コードが間違っているのか」
  を切り分けられなくなる。
- **エラー処理を早い段階(Step 3)で固める**。
  エンドポイントを増やすのは単純作業だが、エラー処理は共通基盤で、
  後から入れると全メソッドを直すことになる。
  逆に先に固めれば、以降のステップは「1 行メソッドを足すだけ」になる。
- **単一オブジェクト → 一覧 → 検索の順**でエンドポイントを増やす。
  レスポンスの形が単純なものから複雑なものへ進む。

## ステップ一覧

### Step 0: API を触る(実装ゼロ)

モックサーバーを起動し、curl で各エンドポイントを叩いて
レスポンスの形を目で確認する。

- ゴール: 実装対象の API を理解する
- 完了条件:
  - `curl -H "Authorization: Bearer x" http://localhost:8793/users/me` で JSON が返る
  - ヘッダーなしだと 401 が返ることを確認した
  - `/folders/0/items` のコレクション形式(`total_count` / `entries`)を見た
  - [api.md](api.md) のエンドポイント一覧・エラー分類に目を通した

### Step 1: 素の HTTP GET(スクリプト 1 本)

クライアントの設計はまだ考えない。その言語の標準 HTTP ライブラリで
`/users/me` を 1 回だけ叩き、レスポンス Body をそのまま表示する
スクリプトを書く。

- ゴール: その言語で「認証ヘッダー付き GET」を送る方法を知る
- 完了条件:
  - Bearer ヘッダー付きで GET し、生の JSON 文字列が表示される
  - ヘッダーを外すと 401 の挙動が観察できる(例外になる言語もある。
    ここでの挙動が Step 3 の設計材料になる)

### Step 2: クライアントの骨格(正常系のみ)

Client クラス(または構造体)を作り、メソッドは `current_user` の 1 本だけ。
JSON をパースしてユーザー名を表示できるところまで。
エラー処理はまだ書かない(200 が返る前提でよい)。

- ゴール: 「設定(base_url / access_token)を持ち、パス指定で GET して
  パース結果を返す」という骨格を固める
- 完了条件:
  - `client.current_user` 相当の呼び出しでユーザー名が取れる
  - base_url の末尾スラッシュを正規化している
  - トークンがクライアント内部で付与されている(呼び出し側に書かせない)

### Step 3: エラー 4 分類(このリポジトリの肝)

エラー型(基底 + 4 分類)を定義し、レスポンス判定処理を実装する。
判定順序: ステータス → 空 Body → JSON パース → 形チェック。
順序の理由は [api.md](api.md) の「判定フロー」を参照。

- ゴール: 異常なレスポンスを 4 分類のどれかに必ず落とせるようにする
- 完了条件(検収):
  - `/broken/http-error` → HTTP エラー(status と body が取れる)
  - `/broken/empty` → 空レスポンス
  - `/broken/truncated` と `/broken/not-json` → JSON パースエラー
  - `/broken/scalar` → 想定外レスポンス
  - 4 分類を呼び出し側で個別に捕捉できる(基底での一括捕捉もできる)

### Step 4: 単一リソース系エンドポイント

`user(id)` / `folder(id)` / `file(id)` を追加する。
パスに ID を埋め込むだけで、Step 2〜3 の基盤がそのまま使える。

- ゴール: 「エンドポイント追加は 1 行」になっていることを確かめる
- 完了条件:
  - `folder("0")` でルートフォルダの名前とサイズが取れる
  - `folder("0")` の `created_at` が null であることを壊れずに扱える
    (静的型付け言語では null 許容の型設計がここで問われる)
  - 存在しない ID(例: `/files/999`)で 404 が HTTP エラーとして捕捉できる

### Step 5: 一覧系エンドポイント(コレクション形式)

`folder_items(id)` / `file_comments(id)` / `folder_collaborations(id)` を追加する。
Box の一覧は `{total_count, entries, offset, limit}` に包まれている。

- ゴール: コレクション形式の扱い方を決める
  (動的型付け言語ならそのまま返す、静的型付け言語なら Collection 型を定義)
- 完了条件:
  - `folder_items("0")` の entries から type / name が取り出せる
  - entries に folder と file が混在していても壊れない
  - 空の一覧(`/folders/0/collaborations`)が正常に扱える

### Step 6: 検索とクエリパラメータ

`search(query)` を追加し、あわせて一覧系メソッドが
任意のクエリパラメータ(`limit` など)を素通しできるようにする。

- ゴール: クエリ文字列の組み立て(エスケープ含む)を正しく行う
- 完了条件:
  - `search("report")` で結果が返る
  - クエリを付けない `/search` が 400 → HTTP エラーとして捕捉できる
  - スペースや日本語を含む検索語がエスケープされて送られる

### Step 7: 仕上げ

- サンプルコード(example)を整備し、全 API 呼び出しと
  エラー分岐の見本をまとめる
- README を書く(使い方、エラー表、動かし方)
- (任意)実際の Box API に Developer Token で接続して動作確認する

- 完了条件: リポジトリの「完成条件」
  (ビルド・実行できる / サンプルが動く / 読み取り系 API が使える /
  エラー処理がある / コード量が少ない)を満たしている

## ステップの飛ばし方

- その言語に多少慣れているなら Step 1 は省略してよい。
- Step 4〜6 は独立性が高いので、必要なエンドポイントから着手してもよい。
  ただし **Step 3(エラー 4 分類)だけは飛ばさない** こと。
  ここを後回しにすると、全メソッドを二度書くことになる。
